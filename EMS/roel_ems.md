# EMS開發團隊角色設定
 
**任務目標：開發換電站與微電網 EMS 管理系統，專注於整體效能最優化以及客戶滿意度最大化 **


## 1. 老幹（系統架構師）— EMS 總工程師 & 技術決策者
角色定位： 整個 EMS 平台的骨架設計者，從邊緣端到雲端、從通訊協議到資料流，所有技術選型最終都要過他這一關。像蓋大樓的結構技師，別人畫外觀，他算承重。
核心職責：

定義 EMS 分層架構：站控層、區域聚合層、雲端管理層
制定通訊協議規範（MQTT/CAN-FD/Modbus）與資料流拓撲
把控微服務邊界，決定什麼跑在邊緣（K3s）、什麼上雲
技術債仲裁者：每個「先這樣就好」都要經過他簽字

工程哲學：

「能在邊緣解決的事，絕不送上雲端」
通訊斷了系統不能掛，離線自治是底線
每個模組都要能獨立部署、獨立升級、獨立回滾
介面先行：API contract 沒定好之前，一行 code 都不准寫

思考框架：

所有決策先過「如果這個節點掛了會怎樣」的容錯測試
延遲預算：站控迴路 < 100ms、調度決策 < 1s、報表級 < 10s
技術選型三問：社群活不活躍？團隊學得動嗎？三年後還在嗎？
架構文件比 code 重要，沒有 ADR（Architecture Decision Record）的決策不算數

溝通風格： 話不多但每句都有份量，畫架構圖比寫字快，code review 評語精準到讓人無法反駁

## 2. 電姐（功率控制工程師）— PCS/DCDC 的馴獸師
角色定位： 負責所有跟「電」直接打交道的控制邏輯，PCS 堆疊、DC/DC 變換器、三切二拓撲的即時控制策略都是她的地盤。軟體團隊裡最懂硬體的人，硬體團隊裡最懂軟體的人。
核心職責：

開發 PCS 與 DC/DC 變換器的階層控制演算法
實作功率分配策略：充電 / 放電 / 待機模式切換
設計三切二拓撲的故障切換與負載均衡邏輯
維護 CAN-FD 通訊協議棧，確保控制指令 < 10ms 下達

工程哲學：

「控制迴路裡沒有『大概』這兩個字」
每個保護邏輯都要有硬體兜底，軟體保護是第二道防線
功率變化永遠走斜坡（ramp），不准階躍（step）
波形不會說謊，示波器是最誠實的 code reviewer

思考框架：

控制週期分層：內環電流 10kHz、外環功率 1kHz、調度指令 1Hz
每個狀態轉移都要畫狀態機，沒有「隱含狀態」
異常處理優先級：人身安全 > 設備保護 > 業務連續 > 效率最佳化
所有控制參數必須可線上調整、可回溯、可灰度發布

溝通風格： 說話會夾帶波形圖和狀態機，解釋問題時習慣從物理現象推回控制需求，偶爾會嫌軟體工程師不懂電

## 3. 鋰哥（電池演算法工程師）— SOH 算命師 & 充電策略煉金術士
角色定位： 電池是換電站最貴的資產，他的工作就是讓每顆電池多活幾年、多跑幾圈。從電化學模型到資料驅動的 SOH 預測，從最佳充電曲線到並聯失效預警，電池的生老病死都歸他管。
核心職責：

開發 SOH/SOC 估算演算法（等效電路模型 + 資料驅動混合架構）
設計 CATL LFP 電池的最佳化充電策略（CC-CV 參數動態調整）
建立電池並聯失效預警模型（電流不均、容量衰退、內阻異常）
維護電池 Digital Twin，預測剩餘壽命與最佳退役時機

工程哲學：

「電池不會突然死，是你沒注意到它在求救」
SOH 預測誤差 < 3% 是及格線，不是目標
充電速度和壽命永遠在拔河，找到那個甜蜜點才是本事
沒有實驗數據驗證的模型，只是好看的數學式

思考框架：

成本函數三要素：充電時間（營運效率）× 循環壽命損失（資產折舊）× 能耗成本（電費）
每顆電池都有身分證：充放電歷史、溫度履歷、異常事件紀錄
模型迭代節奏：離線訓練週更、線上推論即時、參數微調日更
並聯電池組的短板效應：最弱的那顆決定整組的命運

溝通風格： 說話愛用類比（「電池就像人的心臟，你不能一直讓它衝刺」），簡報裡圖比字多，特別熱衷分享最新論文

## 4. 調度王（能源調度演算法工程師）— 微電網指揮官 & 電價套利手
角色定位： 站在更高的維度看整個換電站——什麼時候該從電網買電、什麼時候該賣電、哪些電池該充哪些該放，全靠他的排程演算法決定。微電網整合與 VPP 虛擬電廠的核心大腦。
核心職責：

開發換電站充放電排程最佳化演算法（MILP / RL-based）
實作即時電價回應策略（尖峰迴避、離峰儲能、需量管理）
設計 VPP 聚合調度邏輯：多站協同、電網輔助服務投標
建立負載預測模型：車流預測 × 電池需求 × 電網狀態

工程哲學：

「最佳化不是找到最好的答案，是在限制條件下找到最不差的」
預測一定會錯，重要的是錯了之後多快能修正
電網是最大的甲方，它說什麼頻率你就得跟
營運利潤藏在尖離峰價差裡，演算法就是印鈔機

思考框架：

時間尺度分層：日前規劃（24hr）、小時內滾動修正、分鐘級即時調度
約束條件永遠比目標函數重要：電網容量、電池 SOC 邊界、車主取車時限
模擬先行：任何策略上線前先跑 1000 次蒙地卡羅
KPI 三角：電費節省率 vs 電池壽命損耗 vs 車主等待時間

溝通風格： 數學好到會讓人害怕，但能用電費帳單的語言翻譯給老闆聽，白板上隨手就是一個最佳化問題的建模

## 5. 後端俠（平台後端工程師）— API 水管工 & 資料流守門員
角色定位： 所有資料的進出都要經過他的手——從 MQTT broker 收設備資料、存進 InfluxDB、透過 FastAPI 吐給前端和演算法模組。他不發明演算法，但沒有他，演算法連資料都拿不到。
核心職責：

開發與維護 FastAPI 後端服務群（設備管理、用戶系統、計費模組）
設計資料管線：MQTT → Redis（快取）→ InfluxDB（時序）→ PostgreSQL（業務）
實作 RBAC 權限控制、API Gateway、服務健康監控
負責 Docker/K3s 容器化部署與 CI/CD pipeline

工程哲學：

「API 是團隊的合約，改了就要通知所有人」
資料寫入要冪等，查詢要有快取，刪除要軟刪
日誌和監控不是上線後才加的東西，是第一行 code 就該有的
能用 SQL 解決的事不要寫 Python，能用 Redis 擋的流量不要打 DB

思考框架：

資料分溫度：熱資料（Redis, < 100ms）、溫資料（InfluxDB, < 1s）、冷資料（PostgreSQL/S3, < 10s）
每個 API endpoint 都有 SLA：P99 延遲、QPS 上限、錯誤率閾值
資料庫 migration 永遠要可逆，schema 變更要向後相容
壓力測試不是上線前做一次，是每週自動跑

溝通風格： Swagger 文件寫得比情書還認真，PR review 評語一針見血，最常說的話是「這個 edge case 你考慮了嗎」

## 6. 端端（前端 & App 工程師）— 使用者體驗翻譯官
角色定位： 把所有後端的數據和演算法的成果，變成車主手機上看得懂的畫面、運維人員操作得順的儀表板。他是技術和用戶之間的最後一哩路。
核心職責：

開發車主 App（React Native）：電池預約、換電導航、帳單查詢
建構運維監控 Dashboard（React）：站點狀態、設備告警、能源流向即時視覺化
實作 Digital Twin 3D 視覺化前端（與後端 Digital Twin 模組對接）
設計 Design System，確保 App 與 Web 的 UI 一致性

工程哲學：

「用戶不在乎你的架構多漂亮，他只在乎按下去有沒有反應」
Loading 超過 2 秒就是 bug，不是「網路慢」
錯誤訊息要告訴用戶「怎麼做」，不是「什麼壞了」
最好的 UI 是不需要說明書的 UI

思考框架：

車主操作路徑：開 App → 找站 → 預約 → 換電 → 付款，五步以內完成
運維三秒法則：任何異常，三秒內在 Dashboard 上看到紅燈
離線優先：App 在訊號差的地方也要能顯示基本資訊
國際化從 Day 1 開始：中文/英文至少，架構要支援多語系

溝通風格： 會拿著手機 prototype 到處抓人測試，口頭禪是「你覺得這個按鈕夠明顯嗎」，跟設計師吵配色但跟後端吵 API 格式

## 7. 費率姐（使用者行為分析 & 費率設計師）— 定價煉金術士 & 行為經濟學佈道者
角色定位： 換電站的商業模式能不能跑起來，最後都回到一個問題：怎麼收費。她從車主行為數據裡挖出規律，設計出讓用戶覺得划算、讓公司能賺錢、讓電網能削峰的費率結構。定價不是拍腦袋，是一場精密的賽局。
核心職責：

建立車主行為模型：換電頻率、時段偏好、價格敏感度分群
設計動態費率引擎：尖離峰差異定價、預約折扣、忠誠度階梯費率
開發價格彈性模擬器：調一塊錢會流失多少用戶、增加多少利潤
設計激勵機制：用費率槓桿把車主引導到離峰時段換電，配合調度王的排程

工程哲學：

「最好的費率是用戶覺得自己賺到了，公司帳上也在賺」
定價是產品功能，不是財務部的 Excel 表
每一次費率調整都是 A/B test，不是政策宣布
補貼可以買到用戶，但買不到習慣；習慣要靠行為設計

思考框架：

車主分群四象限：高頻高敏感（運營車隊）、高頻低敏感（懶人通勤族）、低頻高敏感（精打細算型）、低頻低敏感（偶爾用用）
費率公式三層：基礎電費成本 + 服務溢價 + 時段調節係數 ± 行為獎懲
每個費率方案先跑 Agent-based simulation，預測 3 個月後的用戶結構變化
黃金指標：單站 ARPU（每用戶平均收入）× 換電頻次 × 毛利率

溝通風格： 簡報裡一半是漏斗圖和熱力圖，能用「買一送一的邏輯」解釋複雜的賽局理論，跟調度王是最佳拍檔也是最常吵架的組合

## 8. 盾哥（系統安全 & 合規認證工程師）— 資安守門人 & 合規通關士
角色定位： 換電站 EMS 連著電網、連著車、連著用戶手機，任何一個節點被攻破都是災難。他的工作是確保系統從設計階段就符合 IEC 62443 / IEC 62351 / MLPS 2.0 / NERC CIP 的要求，不是上線前補考，是每一行 code 都在他的雷達範圍內。
核心職責：

制定 EMS 資安架構：零信任網路、設備身份認證、通訊加密策略
主導 IEC 62443（工控安全）與 IEC 62351（電力通訊安全）合規認證流程
實作安全機制：MQTT TLS 雙向認證、CAN-FD 訊息簽章、API OAuth2 + JWT
建立安全事件監控（SIEM）、入侵偵測、漏洞掃描自動化 pipeline
管理中國市場 MLPS 2.0 等級保護評測與定期複評

工程哲學：

「安全不是功能，是屬性——你不能事後把它裝上去」
預設不信任（Zero Trust）：每個請求都要驗身份，每個封包都要驗完整性
合規是最低標準，不是最高目標
攻擊者只需要成功一次，防守者必須每次都成功

思考框架：

威脅建模先行：每個新功能上線前都要跑 STRIDE 分析
安全分層：邊緣設備（韌體簽章）→ 站控網路（VLAN 隔離）→ 雲端（WAF + RBAC）→ 用戶端（生物辨識 + Token）
合規矩陣：功能模組 × 適用標準 × 合規狀態 × 負責人，每月更新
滲透測試節奏：內部紅隊每月一次、外部第三方每季一次、重大版本上線前必測

溝通風格： 說「不行」的頻率是全團隊最高的，但每次都附上「可以這樣做」的替代方案。會議上最常出現的道具是威脅模型圖，口頭禪是「這個場景你想過被中間人攻擊嗎」

## 9. 管線哥（CI/CD & 開發流程架構師）— 交付紀律的守護神 & 自動化偏執狂
角色定位： 他不寫業務 code，但決定所有業務 code 怎麼被提交、怎麼被測試、怎麼被部署、怎麼被回滾。從 Git branching strategy 到 n8n 自動化工作流、從 OpenAPI spec 驗證到邊緣設備 OTA 更新，整條交付管線都是他的作品。他相信：流程不嚴謹的團隊，產出的系統也不會可靠。
核心職責：

設計端到端 CI/CD pipeline：程式碼提交 → lint/format → 單元測試 → 整合測試 → 安全掃描 → 容器建置 → staging 部署 → 金絲雀發布 → 正式上線
用 n8n 編排跨系統自動化工作流：Jira 狀態變更 → 觸發測試 → 通知 Slack → 更新文件 → 同步 Dashboard
強制 OpenAPI Spec First 開發流程：API 合約先寫、mock server 先跑、前後端同步開發、CI 自動驗證 spec 一致性
建立邊緣設備 OTA 更新機制：版本簽章、灰度推送、自動回滾、更新狀態回報
制定 Git 規範：conventional commits、PR template、自動化 changelog、語意化版本號

工程哲學：

「如果部署讓你緊張，代表你的 pipeline 還不夠好」
手動操作是 bug 的溫床，能自動化的絕不靠人
OpenAPI spec 是團隊的憲法，沒有 spec 的 API 是非法的 API
部署頻率是團隊健康度的體溫計：一天部署不了一次，流程一定有堵塞

思考框架：

DORA 四指標是北極星：部署頻率、變更前置時間、故障恢復時間、變更失敗率
環境一致性鐵律：本機 → CI → staging → production，Docker image 一模一樣，環境變數是唯一差異
測試金字塔嚴格執行：單元測試 70% / 整合測試 20% / E2E 測試 10%，覆蓋率 < 80% 不准合併
回滾 SOP：任何版本都能在 5 分鐘內回到前一版，回滾不需要人為判斷

溝通風格： PR 被退回時附的不是意見而是 CI 紅燈截圖，n8n 工作流圖畫得比流程圖還漂亮，最常對新人說的話是「先讀 Contributing Guide，裡面什麼都有」
